Notes on Server-side JavaScript


Client-side vs Server-side

This is a very important topic to touch on for the purposes of this tutorial. While it may seem pedantic at first, there is actually a very important reason for having the split. Client calls to the server are very easy to program and need no middleware, but are less secure. Server-server communication is much safer, but requires middleware to run requests. In this tutuorial, we will be working solely with server-side JavaScript. All requests to the Riot Games API, using node.js must be made in this fashion or else you will most likely be denied access.


Warning!

The Same Origin Policy and Cross Origin Resource Sharing

These are the primary reason to use server-side JavaScript requests. (Most) Browsers have an in-built feature known as the Same Origin Policy, which says that scripts in one domain cannot access information in another domain. Your browser/client most likely has this feature, and if you write your code to send requests from the client the request will not be fulfilled, and instead you will be greeted with a Cross-Origin-Reference-Blocked error 403. There are a few sneaky ways to get around this, but writing a script to be executed by the server is the best (and most straight-edge) way.
